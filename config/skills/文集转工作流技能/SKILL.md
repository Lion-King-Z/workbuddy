# 文集转工作流技能

## R — 核心原则

> 把文集变成可执行技能，关键不是按主题分类，而是按“决策节点”分类。每个 skill 必须是一个连续的工作流，解决一个具体的投资/工作问题。
>
> — 基于常总知识库 21 篇投资哲学文章的蒸馏实践

---

## I — 方法论骨架

传统 book2skill 适合单本结构化的书。文集没有主从章节，主题重叠，按主题拆分会产出大量细碎、难调用的 skill。本方法先把文集按“用户决策节点”重新聚类，再把每个节点下的内容写成单一工作流 skill。每个工作流由 3-6 个有序步骤构成，每一步都有完成标准和判停条件，最终输出一个可执行的结论或建议。

---

## A2 — 触发场景

1. 用户给出一个目录下多篇相关文章，要求提炼为 skill。
2. 用户抱怨按主题拆出的 skill 太细、不会用。
3. 用户明确要求“工作流”“一步一步做”“不要模块化”。

语言信号：
- “把这几篇文章做成 skill”
- “不要模块化”
- “按工作流来”
- “太细了，合并一下”

---

## E — 可执行步骤

### 步骤 1：通读并列出所有文章标题

- 完成标准：拿到全部文章列表，不遗漏。
- 判停条件：如果用户只给了一个主题但未指定路径，先询问文集路径。

### 步骤 2：识别决策节点

- 完成标准：从文章标题和内容中提炼出 3-5 个投资决策/工作节点，而不是主题类别。
- 示例：
  - 不是“超级赛道选择”这种主题，而是“这个赛道/公司值不值得投”这个决策问题。
  - 不是“估值方法”这种主题，而是“现在价格贵不贵、能不能买”这个决策问题。
- 判停条件：如果节点超过 6 个，说明分类太细，合并到 3-5 个。

### 步骤 3：把文章分配到决策节点

- 完成标准：每篇文章只归入一个主节点；有重叠时按“最核心解决的问题”归类。
- 判停条件：如果一篇文章同时服务多个节点，放在最前置的节点或拆分到两个节点的边界中说明。

### 步骤 4：为每个节点写一个单一 SKILL.md

- 完成标准：每个 skill 文件包含：
  1. R：原文引用（≤150 字）
  2. I：工作流整体说明（5-15 行）
  3. A1：2-3 个应用案例
  4. A2：触发场景 + 语言信号 + 与相邻 skill 的区分
  5. E：3-6 个连续执行步骤，每个步骤有完成标准、输入、输出、判停条件
  6. B：边界、失败模式、作者盲点、邻近方法论区别
- 强制约束：
  - 不得出现子 skill
  - 不得把内容拆成“框架/原则/案例”等独立模块
  - 所有方法论必须融入同一条工作流主线
- 判停条件：若某个节点内容太少无法形成独立工作流，与相邻节点合并。

### 步骤 5：生成 test-prompts.json

- 完成标准：每个 skill 有一个 test-prompts.json，包含 should_trigger / should_not_trigger / edge_case 三类各至少 3 条。
- 判停条件：边界不清时补充 edge_case。

### 步骤 6：生成 INDEX.md

- 完成标准：INDEX 中说明：
  - 每个 skill 的功能和触发词
  - 投资决策流程图（mermaid）
  - 与现有 skill 的协作关系（不修改原有 skill）

### 步骤 7：安装到用户 skills 目录

- 完成标准：
  - 复制到 `~/.workbuddy/skills/`
  - 如果存在旧的模块化版本，可选择性移除（需确认用户意图）
  - 不修改用户原有其他 skill
- 判停条件：安装完成后验证目录存在。

---

## B — 边界

- 不要用于单本结构化书籍的拆分；那种情况用仓颉Skill。
- 不要用于修改/重排用户已有技能体系；只在用户明确授权时移除旧版本。
- 不要产出超过 5 个 skill；超过说明节点切分过细。
